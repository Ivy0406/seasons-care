# 前端 Code Review

## 目標
- 以協作為前提，優先找出會影響功能、穩定性、維護性與整合風險的問題。
- 盡量避免只基於個人偏好的意見。
- 盡量提出低衝突、可落地、可驗證的建議。

## 審查順序

1. **功能與回歸風險**
   - 是否會造成邏輯錯誤、錯誤資料流、狀態不同步、頁面行為異常
   - 是否可能破壞既有功能或使用情境

2. **型別與執行期安全**
   - 型別是否過鬆、缺少 null / undefined 防護
   - 是否可能在執行期拋錯
   - 非同步流程是否有漏接錯誤

3. **整合與協作風險**
   - 是否違反目前專案既有資料流、共用元件用法或 API 契約
   - 是否容易和其他分支衝突
   - 是否在不必要的地方引入大範圍改動

4. **效能與可維護性**
   - 是否有明顯不必要重算、重複渲染、重複請求
   - 命名、拆分、邏輯層次是否清楚

5. **安全性**
   - 是否有明顯 XSS、注入、敏感資訊暴露或不安全資料使用方式

## 協作優先原則
- 先講高風險問題，再講改善建議。
- 只提和目前修改範圍直接相關的問題。
- 避免要求為了 review 結果進行大規模重構，除非現在這版明顯不安全或不可維護。
- 若有多種修法，優先建議與現有模式一致、衝突最小的做法。
- 若沒有明確問題，就直接說沒有發現阻擋性問題，不要硬挑小毛病。

## 不要聚焦的低價值項目
- 純個人偏好的命名意見
- 和此次修改無關的歷史舊問題
- 沒有實際風險證據的過度設計建議
- 只為了「更乾淨」而提出的大範圍重構

## 輸出格式

先列出 findings，依嚴重度排序，並附上檔案位置。

### 必須修正
- `[檔案:行號] 問題描述`
  - 風險
  - 建議做法

### 建議改善
- `[檔案:行號] 問題描述`
  - 原因
  - 建議做法

### 開放問題
- 需要確認但目前無法單靠程式碼判定的點

### 總結
- 一到三句總結整體狀況

## 額外規則
- 若沒有發現明確問題，請直接寫：

```md
### 必須修正
- 未發現阻擋合併的明確問題
```

- 若是 review PR 或 diff：
  - 以本次變更為主
  - 不要把整個專案重新審一次

- 若是 review React / 前端程式碼：
  - 特別檢查 state 流向、effect 使用、受控元件、非同步請求、條件渲染與共用元件 API 是否一致
